pi-context-tree — remove a whole Q&A turn (mockup for review)

Your friend's point, made concrete: you select a question + its answers and they drop together. Removing only the answer would leave a dangling question and orphan tool_call / tool_result pairs — the model breaks on that. Nothing is deleted from disk: the turn moves out of live context via the same branch-at-anchor + reconstruction-block mechanism as /crop; originals stay recoverable on the previous branch.

WHY WHOLE TURNS, NOT SINGLE MESSAGES
A turn = one user question + every assistant / tool entry it spawned, up to the next question. Dropping the set together keeps the remaining context well-formed. Dropping only the answer would orphan a tool_call from its tool_result and leave a question with no reply — exactly the corruption your friend described.

HOW IT'S APPLIED  (identical to /crop, just a turn-sized selection)
branch at the anchor (parent of the earliest removed turn) → one ctree/crop-tail reconstruction block: survivors verbatim, each removed turn collapsed to a [dropped turn: …] note → a ctree/crop marker records the removed entry ids + a sha for recovery. The original JSONL is never mutated.

DECISIONS (shipped)
1. Entry point → a t toggle inside the crop panel (result ⇄ turn). Turn removal is panel-only — a "pick this specific exchange" action, not a bulk rule.
2. Drop note → kept, but label-free: [dropped turn — N entries, ~tokens, recoverable: sha8]. Echoing the question would re-inject what you removed; the readable label lives in the ctree/crop marker for the panel.
3. Decision-record turns → protected by construction: a ◆ record is a turn boundary, never a turn member, so it can't be selected or swept up. Bonus: the current/leaf turn is protected too.
